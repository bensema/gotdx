# gotdx 项目规则 (RULES)

本文档定义 `gotdx` 项目的开发、贡献与 AI 改码约束。目标是让规则和当前仓库保持一致，优先保障：

- Go SDK 的可维护性
- 二进制协议实现的正确性
- 对外 API 的稳定性
- 示例、测试、文档三者同步

## 1. 项目定位

- `gotdx` 是一个面向通达信行情协议的 Go SDK，不是重量级框架。
- 根目录文件负责对外公开 API 与客户端封装，例如 `client.go`、`client_quote.go`、`client_unified.go`。
- `proto/` 负责底层协议报文、序列化、反序列化和协议相关测试。
- `examples/` 提供终端用户可直接运行的示例。
- `cmd/webviewer/` 是仓库内附带工具，不应反向污染 SDK 主体设计。

## 2. 语言与代码风格

- 核心语言为 Go，遵循 Go 官方习惯用法，优先可读性与一致性。
- 所有代码必须通过 `gofmt`；如调整 import，使用 `goimports` 保持分组整洁。
- 命名遵循 Go 约定：
  - 导出符号使用大写开头。
  - 非导出符号使用小写开头。
  - `API`、`CSV`、`TCP`、`URL` 这类常见缩写保持 Go 社区常见写法，不混用奇怪大小写。
- 新增导出类型、函数、方法时，应补充简洁的 Go 风格注释，说明用途而不是重复签名。
- 文件职责应单一；不要在一次提交里混入无关重构、批量改名或格式噪音。

## 3. API 与目录约束

- 公开 API 优先保持兼容；修改导出函数签名、导出结构字段或返回语义时，必须同步更新：
  - `README.md`
  - 相关示例
  - 相关测试
- 新增主行情能力时，优先按现有模式落在根目录 `client_*` 文件中；扩展市场能力沿用 `client_exquote.go` / `client_unified.go` 的组织方式。
- 底层协议细节应收敛在 `proto` 包，不要把报文编解码逻辑散落到上层客户端。
- 示例代码应继续复用 `examples/internal/exampleutil/client.go` 的连接方式，避免每个示例重复维护主机列表。
- 新增协议能力时，禁止只补单点实现；必须按仓库当前对外形态把整条链路补齐：
  - `proto/` 中的请求、响应、序列化、反序列化
  - 根目录对外 client API，例如 `Get*`、`ExGet*`、`Stock*`、`Ex*`、`MAC*`
  - `cmd/webviewer/query.go` 中的 method 定义、参数、返回展示
  - `examples/` 中至少一个可直接运行的对应示例
  - `README.md` 中的能力说明或示例清单

## 3.1 新增协议标准操作模板

凡是“新增一个 protocol / method / 协议号”，默认都要完成下面这套最小闭环，除非任务明确说明只做草稿或只做底层验证：

1. `proto` 层补齐
   - 新增或更新协议结构体、请求参数、响应结构、`Serialize`、`UnSerialize`
   - 新增对应单元测试，至少覆盖一个序列化断言和一个反序列化断言
   - 若协议属于兼容/试验协议，也要保证返回值可观测，必要时至少返回原始字节/hex

2. client 层补齐
   - 在合适的 `client_*.go` 中补底层调用入口
   - 如项目已有统一高阶封装模式，再在 `client_unified.go` 或对应聚合入口中补上统一方法
   - 导出能力时补简洁注释，名称遵循现有模式，不要自行发明新风格

3. web viewer 补齐
   - 在 `cmd/webviewer/query.go` 中补 method 元信息
   - 补参数解析、调用逻辑、返回表格列和原始数据展示
   - 如果是试验协议，至少要能在页面里看到可用结果，而不是只有“已实现”但无法验证

4. examples 补齐
   - 在 `examples/` 下新增或更新一个能直接运行的示例
   - 示例输出应以终端用户可读为目标，避免只打印不可读的大对象
   - 如已有相近示例，可在原示例中自然并入；如是独立能力，优先单独建目录

5. README 补齐
   - 在 `README.md` 的示例清单、能力概览或 API 速览中补上相关入口
   - 文档描述必须与代码真实能力一致，不能出现代码已加但 README 缺失，或 README 已写但代码不可用

6. 验证收尾
   - 执行 `gofmt`
   - 执行 `go test ./...`
   - 若协议依赖真实服务器验证，记录“是否已联网验证、哪些 host 成功/失败、失败更像协议问题还是 host/IP 问题”

可直接复用的检查清单：

- [ ] `proto` 已补
- [ ] `proto` 测试已补
- [ ] client API 已补
- [ ] `cmd/webviewer/query.go` 已补
- [ ] `examples/` 已补
- [ ] `README.md` 已补
- [ ] `go test ./...` 已通过

## 4. 依赖策略

- 优先使用标准库；新增第三方依赖必须有明确收益，不能只是为了省几行代码。
- 禁止引入重量级网络框架、序列化框架或与当前项目体量不匹配的基础设施依赖。
- 当前仓库已使用 `golang.org/x/text`，类似依赖只应在确有协议或字符集处理需求时保留。

## 5. 协议、网络与性能

- 协议正确性优先于“想象中的极致优化”。
- 在 `proto` 包中处理定长/二进制字段时，优先使用明确、静态的字节读写方式，例如 `binary.LittleEndian`。
- 对网络收发、解压、报文解析这类热点路径，应避免明显多余分配；但 `sync.Pool`、对象池、复用型优化不是默认强制项，必须在热点明确、收益可解释时再引入。
- 当前 `Client` 的单连接请求收发以互斥锁串行化为基础。新增功能时：
  - 不得破坏现有调用安全性。
  - 不能引入 data race。
  - 若扩展为连接池、自动重连或更复杂并发模型，必须先定义清楚语义并补测试。
- 超时、地址池、主/扩展市场地址切换等行为，应继续通过 `Options` / `With*` 选项暴露，不要写死到业务逻辑中。

## 6. 错误处理

- 不要吞掉真实业务错误；返回给调用方时附加足够上下文，例如 `fmt.Errorf("read response header: %w", err)`。
- 对于输入校验或稳定可识别的错误，优先定义导出错误变量，例如 `client_helpers.go` 中的 `ErrMarketCodeCount`。
- 类似 `Close`、`SetDeadline` 这类 best-effort 清理动作，如果忽略错误，必须确认不会影响主流程语义。
- 库代码默认不使用 `panic` 处理普通错误。

## 7. 测试与验证

- 默认要求 `go test ./...` 可通过，且不依赖外部网络环境。
- 联网测试必须默认跳过，并由 `GOTDX_INTEGRATION=1` 显式开启；当前约定已在 `client_test.go` 中使用。
- 新增 `proto` 编解码逻辑时，优先补充纯单元测试，使用固定字节流验证序列化和反序列化结果。
- 修复 bug 时，优先先补能复现问题的测试，再修实现。
- 若修改 `cmd/webviewer/` 的查询映射、参数解析或输出格式，应同步补对应测试。
- 若新增协议后未补 `proto/client/webviewer/examples/README` 中任一关键环节，应视为未完成，而不是“后续再说”。

## 8. 示例与文档

- 新增公开能力时，至少满足以下之一：
  - 补充或更新对应示例
  - 在 `README` 中补上可用说明
- 示例必须能被终端用户直接理解和运行，避免把内部调试代码、临时打印、半成品逻辑留在 `examples/`。
- 若行为变更会影响用户用法，必须更新 `README.md`，不能只改代码不改文档。
- 对“新增协议”类任务，上面两条不是二选一，而是默认都要做：示例要补，`README.md` 也要补。

## 9. AI 改码规则

- 先观察现有模式，再落地修改；优先复用仓库已有命名、错误风格、返回风格和文件组织。
- 不得凭空发明仓库中不存在的架构层、接口层或配置系统。
- 不得为了“看起来更高级”引入与项目规模不匹配的抽象。
- 不得把建议写成事实；文档、注释、README 中提到的能力必须在代码中真实存在。
- 新增代码时，优先做最小必要改动；非任务相关的格式化、重排、重命名不要顺手混进来。

## 10. 提交前检查

提交或合并前至少确认：

- 代码已格式化
- `go test ./...` 通过，或明确说明未通过原因
- 新增/修改的公开能力已有测试、示例或文档覆盖
- 没有引入无用依赖、无用导出符号和无意义日志
- 规则、README、示例与代码实现一致
- 如果本次改动涉及新增协议，已按 `3.1 新增协议标准操作模板` 完成 `proto + client + webviewer + examples + README` 全链路补齐
