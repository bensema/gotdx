# RULES

`RULES.md` 记录的是 `gotdx` 项目的工程约束，而不是 AI 的行动流程。

- `AGENTS.md` 负责回答“怎么做事”
- `RULES.md` 负责回答“什么不能做错”

如果两者有重叠，以 `AGENTS.md` 的执行流程为入口，以 `RULES.md` 的工程约束为落地边界。

对新进入仓库的 agent，默认先读 `AGENTS.md`，再回到本文件确认工程边界。

## 1. 项目定位与边界

- `gotdx` 是面向通达信行情协议的 Go SDK，不是重量级应用框架。
- 优先保障协议正确性、对外 API 稳定性、示例可运行性和文档一致性。
- `cmd/webviewer/` 是仓库自带工具，但不能反向主导 SDK 的核心设计。

## 2. 目录职责

- 根目录 `client*.go`: 对外公开 API、连接逻辑、统一高阶封装
- `proto/`: 协议结构、请求/响应编解码、协议相关测试
- `examples/`: 面向终端用户的可运行示例
- `cmd/webviewer/`: 浏览器侧调试入口、方法注册、参数解析、结果展示
- `docs/`: 补充说明、协议对照、截图等非核心实现文档

新增代码应放在最自然的目录里，不要让职责漂移。

## 3. API 稳定性

- 公开 API 优先保持兼容。
- 修改导出函数签名、导出结构字段或返回语义时，必须同步更新：
  - `README.md`
  - 相关示例
  - 相关测试
- 新增导出类型、函数、方法时，应补简洁的 Go 风格注释。

## 4. 新增协议的最小闭环

凡是新增一个协议 / method / 协议号，默认必须补齐整条链路：

1. `proto/`
   - 请求结构
   - 响应结构
   - `Serialize`
   - `UnSerialize`
   - 对应测试
2. `client`
   - 低层 `Get*` / `ExGet*` / `GetMAC*`
   - 若已有统一封装模式，再补 `Stock*` / `Ex*` / `MAC*`
3. `cmd/webviewer/query.go`
   - method 元信息
   - 参数解析
   - 实际调用
   - 返回展示
4. `examples/`
   - 至少一个可运行示例
5. `README.md`
   - 能力说明、API 速览或示例清单

如果只补其中一段链路，默认视为未完成，除非任务明确说明只做底层验证。

## 5. 协议实现规则

- 二进制协议正确性优先于表面上的“代码优雅”。
- 编解码逻辑应尽量收敛在 `proto/`，不要散落到上层 client。
- 对定长字段、字节布局、端序处理，应使用明确静态的实现方式。
- 兼容协议或试验协议如果暂时无法完整解析，至少要返回可观测结果，例如原始字节或 hex。

## 6. Client 与连接行为

- 超时、地址池、主站/扩展站切换等连接行为，应继续通过 `Options` / `With*` 选项暴露。
- 不要把 host、端口、超时等网络策略硬编码到业务逻辑里。
- 当前 `Client` 默认基于单连接串行请求模型；新增能力时：
  - 不得引入 data race
  - 不得破坏现有调用安全性
  - 若扩展为更复杂的重连、连接池或并发模型，必须先定义清楚语义并补测试

## 7. 示例规则

- 示例必须面向终端用户可读，不要保留半成品调试代码。
- 示例默认复用 `examples/internal/exampleutil/client.go`。
- 不要在多个示例里各自维护分叉的 host 列表或重复连接逻辑。
- 新增公开能力时，示例和 README 不是二选一，而是默认都要补。

## 8. Web Viewer 规则

- `cmd/webviewer/query.go` 中新增 method 时，必须包含：
  - 明确的 method key
  - 参数定义与默认值
  - 实际调用逻辑
  - 可读的返回字段展示
- 不能只注册一个 method 名称而没有可验证结果。
- 输出应优先便于人工排查协议和字段问题。

## 9. 测试与验证

- 默认要求 `go test ./...` 可通过，且不依赖外部网络。
- 联网测试必须默认跳过，并通过 `GOTDX_INTEGRATION=1` 显式开启。
- 新增 `proto` 编解码逻辑时，优先补固定字节流的纯单元测试。
- 修复 bug 时，优先补复现测试，再修实现。
- 修改 Web Viewer 的参数解析或返回格式时，应同步补对应测试。

## 10. 错误处理

- 不要吞掉真实业务错误。
- 返回错误时尽量附加上下文，便于判断是序列化、网络、解压还是字段解析问题。
- 对稳定可识别的输入错误，优先使用清晰可复用的错误变量。
- 普通库代码默认不使用 `panic` 处理业务错误。

## 11. 依赖与抽象

- 优先使用标准库。
- 新增第三方依赖必须有明确收益，不能只是为了省几行代码。
- 禁止引入与当前项目体量不匹配的重量级框架或抽象层。
- 不要为了“看起来更高级”新增仓库中本来不存在的架构。

## 12. 代码风格

- 遵循 Go 官方习惯，优先可读性和一致性。
- 所有代码必须通过 `gofmt`；如需整理 import，可使用 `goimports`。
- 命名遵循 Go 常见约定，例如 `API`、`TCP`、`URL` 等缩写保持常见写法。
- 文件职责尽量单一，不要在一次改动里混入无关重排或批量重命名。

## 13. 提交前检查

提交或合并前至少确认：

- 代码已格式化
- `go test ./...` 已通过，或已明确说明未通过原因
- 新增或修改的公开能力已有测试、示例和文档覆盖
- README、示例、Web Viewer 与代码实现一致
- 若本次涉及新增协议，已完成 `proto + client + webviewer + examples + README` 全链路补齐
